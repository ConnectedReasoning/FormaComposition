# schemas.py / generator.py — Refactoring Recommendations

Source of this review: `main.py` + `schemas.py` walkthrough, cross-checked against `lint.py`, `generator.py`, `strategies.py`, and `rhythm.py` (grep-verified, not assumed). Item 1 was confirmed against a real render (`piece_broadway_boogie_v7.json`, screenshot of the Harmony track around the `m1_grid_3` → `m2_accumulation` boundary), then corrected after a full read of `generator.py` revealed a second, independent arc implementation (`velocity_envelope`, melody-only) that clarified what was actually a bug versus intended design. Ordered by priority, not file order.

---

## P0 — Fix before sharing with anyone else

### 1. Harmony's arc implementation is missing three real curves that already exist elsewhere in the codebase — CONFIRMED LIVE BUG, audible in an actual render

- **Correction from an earlier version of this item:** originally diagnosed as "`plateau` silently no-ops and shouldn't." That's wrong — `generator.py`'s `velocity_envelope()` (171–211), the *melody* path's independent arc implementation, treats `plateau` as **intentionally** flat (`m = 1.0`), with an explicit comment confirming it: *"Multiplier 1.0 (plateau / unknown arc) is a strict no-op so existing output is unchanged."* Flat is correct for `plateau`. The real bug is narrower: `build`, `decay`, and `fade` have genuine, distinct curves in `velocity_envelope` (`build`: 0.70→1.20 quadratic; `decay`: 0.95→0.70 linear; `fade`/`fade_out`: 1.00→0.65 linear) and are silently flattened in harmony's `apply_velocity_arc` (`rhythm.py:814–838`), which only implements `flat`, `swell`, `fade_in`, `fade_out`, `breath`.
- **Verified against a real piece:** `piece_broadway_boogie_v7.json`'s `m1_grid_3` section uses `arc: "fade_in"` (correctly implemented in both paths) immediately followed by `m2_accumulation` using `arc: "plateau"` (correctly flat, by design — not a bug). The audible seam at that boundary is real and still worth fixing (see item 9, cross-section continuity), but it is not caused by an unimplemented `plateau`. The genuine implementation gap (`build`/`decay`/`fade`) hasn't yet been caught in an observed render, but any past piece using those three values on a harmony-bearing section has silently rendered flat regardless.
- **Compounding factor:** `harmony`'s `base_velocity` is a hardcoded constant (`65`, `strategies.py:955`), never overridden per section — so `arc_scale` is the *only* source of velocity variation for harmony. When `build`/`decay`/`fade` silently no-op, the section has literally zero dynamic shaping, not just reduced shaping.
- **Fix:** port `velocity_envelope`'s existing curve math for `build`, `decay`, and `fade` into `apply_velocity_arc` — the correct curves already exist and are proven out in the melody path, so this is a smaller, lower-risk fix than inventing new ones. See item 8 below for the larger question this raises (two independent, differently-designed arc engines) before deciding exactly how to port them.
- **Why `lint.py` didn't already catch this — a gap in what kind of check the file even knows how to write:** every one of `lint.py`'s nine checks follows the same shape — "field A is only consumed when field B has some other value" (a *coupling*). `arc: "build"` isn't gated by another field; it's a value the type system accepts that one of its two consumer functions never implemented a branch for. That's a structurally different failure — "accepted but unimplemented in this particular consumer," not "ignored due to another setting" — and none of the nine existing checks have a shape that could catch it. Worth adding as a new check category (e.g. `_check_unimplemented_enum_values`, cross-referencing each `Literal`'s members against what its consumer function's `if`/`elif` chain actually branches on) rather than trying to force-fit it into the coupling pattern. This is more mechanical to write than the existing checks — it wants to introspect a function's actual branches, not just read section state — but it's exactly the class of bug that got through silently here.

### 2. `SongFormEntryModel.variation` is a silent, unflagged no-op

- **Where:** `schemas.py` line 953 — declared, typed, range-validated (`0.0–1.0`).
- **Verified:** `grep -n "variation" generator.py` returns zero consuming references — no `_apply_variation`, no `entry.variation` read anywhere. `display_info()` in `main.py` prints it (line 63), which makes it *look* wired.
- **Why it's worse than average:** every other known-dead field in this codebase (`note_duration`, `voice.motif` on non-lead voices, `section.motif`/`motifs`) has a corresponding check in `lint.py`'s `CONTRADICTION_CHECKS` list. `variation` has none — `grep -n "variation" lint.py` returns nothing. A composer can set `"variation": 0.7` on a form entry, see it echoed back in `--info`, get zero lint warnings, and get a byte-identical render to `variation: 0.0`.
- **Fix:** either wire it into the song-form expansion in `generator.py` (design decision: what should "variation" actually vary — velocity? note choice? transform selection?), or add a `_check_form_variation` lint rule modeled on `_check_section_motif_override` so it's at least visible until you decide. Don't leave it silently validated with no lint coverage — that's the one state this codebase has otherwise been disciplined about avoiding.

### 3. `_KNOWN_KEYS` (schemas.py:465–473) is a hand-maintained shadow of `SectionModel`'s real field list

- **Risk:** every time a field is added to `SectionModel`, it must *also* be added to `_KNOWN_KEYS`, or the unknown-key warning (`_warn_unknown_keys`) starts false-firing on a perfectly valid field the moment someone uses it. This is exactly the kind of duplicated-source-of-truth problem the `VALID_DENSITY = set(get_args(DensityLiteral))` pattern elsewhere in this file was built to eliminate — `_KNOWN_KEYS` just wasn't brought in line with that pattern.
- **Fix:** derive it: `_KNOWN_KEYS = set(SectionModel.model_fields.keys())` (computed once, likely in a `model_validator` or as a class method, since it can't reference `SectionModel` from inside its own body). Keeps unknown-key detection meaningful without manual upkeep.

## P1 — Real inconsistencies worth resolving, not urgent

### 4. Two incompatible representations of "a motif"

- `MotifModel` (schemas.py:380–422) is the theme's rich, validated shape: `intervals`, `rhythm`, `rests`, `velocities`, `transform_pool`, with cross-field length checks.
- `VoiceModel.motif`, `HarmonyRhythmModel.motif`, `SectionModel.motif`/`motifs` are all typed as bare `Union[str, dict]` — a string library reference or an untyped dict. The dict form gets zero validation until `motif_loader.resolve_motif_value()` runs at generation time (or during the `validate_against_theme` cross-check, via `_resolve_motif_value_safe`), well outside Pydantic's normal error-surface.
- **Consequence:** a malformed inline motif dict on a voice override won't fail fast at `PieceModel.model_validate()` the way a malformed theme motif would — it fails later, wrapped in a re-raised `ValueError`, at a different point in the pipeline. Two different failure timings and error paths for what is conceptually the same object.
- **Fix:** consider typing the dict-form branch as `MotifModel` instead of raw `dict` (`Union[str, MotifModel]`), so overrides get the same validation timing and error quality as theme motifs. If the looser typing is intentional (e.g. to allow partial/shorthand motif dicts that `motif_loader` fills in), document that asymmetry explicitly next to `MotifModel`'s docstring — right now it reads as an oversight, not a decision.

### 5. `SectionModel.melody: Union[MelodyLiteral, "VoiceModel"]` overlaps with `voices[]`

- Two ways to say the same thing exist by design (`lead_voice()` and `melody_behavior()` both branch on `self.voices` first, `melody`-as-`VoiceModel` second, bare string last) — that's a reasonable back-compat shim, not a bug. But it means every future feature added to `VoiceModel` (like the `motif` override) has to be manually re-checked against *both* entry points, and the "lead voice" concept is scattered across two helper methods that both have to be kept in sync with which branch wins.
- **Fix (low urgency, worth doing before the next voice-related feature):** collapse to a single normalization step — e.g. a `@model_validator(mode="after")` on `SectionModel` that rewrites a `VoiceModel`-form `melody` into `voices[0]` at parse time (when `voices` is absent), so downstream code only ever has one shape to reason about. `lead_voice()`/`melody_behavior()` become trivial once there's one canonical location.

### 6. `note_length_range` doesn't follow this codebase's own inheritance pattern, and `voices[]` is missing it entirely

- **The established pattern:** `DrumModel.resolve(section_density, section_groove, section_swing)` formalizes "child field `None` → inherit section value, child field set → override" as an explicit method. `VoiceModel.rest_probability` follows the same convention (`Optional`, docstring: "overrides section default when set").
- **`note_length_range` doesn't follow it:**
  - `SectionModel.note_length_range` — applies to the legacy melody path.
  - `CounterpointModel.note_length_range` — per-voice, docstring says "when None it inherits the section's note_length_range," but that inheritance is asserted in prose, not implemented as a `.resolve()`-style method the way `DrumModel` does it.
  - **`VoiceModel` — the flat `voices[]` peer-array architecture, i.e. the current active architecture — has no `note_length_range` field at all.** A peer voice generated via the behavior path (`melody.py`) has no way to get its own duration range independent of the section. This is a real feature gap in the newer architecture relative to the older `counterpoint` path, not just a style inconsistency.
- **Fix:**
  1. Add `note_length_range: Optional[NoteLengthRangeModel] = None` to `VoiceModel`, meaning "inherit section's."
  2. Give `NoteLengthRangeModel` (or its callers in `generator.py`) an explicit `resolve(section_range)` method mirroring `DrumModel.resolve()`, so the inherit-or-override logic is code, not docstring convention, across all three owners (`SectionModel`, `CounterpointModel`, `VoiceModel`).
  3. **Do not** extend this to harmony or bass wholesale. Harmony's `sustain`/`motif`/`pattern` rhythm sources already fully determine duration — a sampled range would be a fourth inert case for `lint.py` to flag. If anywhere in harmony deserves it, it's narrowly `harmony_rhythm.rhythm == "free"`, the one source structurally analogous to melody's free/generic duration sampling — gate it there specifically, not on the whole `HarmonyRhythmModel`. Bass should stay untouched entirely: `bass_rest_probability` is already documented as ignored for `walking`/`melodic` because those styles derive duration from the walking algorithm's own stepwise logic — a duration range there wouldn't just be inert, it would actively fight the generation logic, which is a stronger objection than the usual no-op case.

### 7. `VoiceModel.motif` is schema-legal on every peer voice but only consumed for the lead voice

- This one's already correctly flagged by `_check_voice_motif` in `lint.py` for the *behavior* gate (motif ignored unless `behavior="develop"`), but that check doesn't address the separate *position* gate — a non-lead voice with `behavior="develop"` **and** its own `motif` set will still have that motif ignored, per the `VoiceModel` docstring, and lint doesn't call that out specifically as a position problem (it reads as a behavior problem in the message).
- **Fix:** either extend `_check_voice_motif` to distinguish "wrong behavior" from "right behavior, wrong voice position" (better error message, same mechanism), or finish wiring per-voice motif for all peer voices, since the architecture (each voice already reads snapshots of previously generated voices) doesn't obviously prevent it — worth a scope check before committing to either.

### 8. Two independent, differently-designed arc implementations exist — `velocity_envelope` (melody) and `apply_velocity_arc` (harmony) — and they don't agree

- **Where:** `generator.py:171–211` (`velocity_envelope`) shapes melody note velocity per bar; `rhythm.py:814–838` (`apply_velocity_arc`) shapes harmony chord velocity per onset. Both exist to implement the same `ArcLiteral` concept, built independently, and now disagree on multiple axes:
  - **Coverage:** `velocity_envelope` implements all 8 `ArcLiteral` values with real, distinct curves for `build`/`decay`/`fade`; `apply_velocity_arc` implements 5 (item 1's bug).
  - **Multiplier range:** `velocity_envelope` produces a 0.6–1.25 multiplier applied on top of a declared velocity; `apply_velocity_arc` produces a 0.5–1.1-ish `arc_scale` (varies by curve) multiplied directly into `base_velocity`. Different shapes of curve, different headroom assumptions.
  - **Clamp bounds:** `velocity_envelope` clamps to `VELOCITY_CLAMP_MIN/MAX` = 40–120 (`generator.py:167–168`); `apply_velocity_arc` clamps to a hardcoded 20–127 inline (`rhythm.py:841`). Two different floors and ceilings for what is conceptually the same "how loud can an arc-shaped note get" question.
- **Consequence beyond item 1:** even after porting `build`/`decay`/`fade` into `apply_velocity_arc` (item 1's fix), melody and harmony will still shape the *same* arc value with *different* curve math and *different* clamp ranges — a `"swell"` section will swell differently for its melody than for its harmony bed, which may or may not be musically desirable but is currently an accident of two people (or two sessions) solving the same problem twice rather than a deliberate choice.
- **Fix:** decide whether melody and harmony *should* share one arc-shaping function (consolidate into a single `velocity_envelope`-style implementation parameterized by track, used by both call sites) or whether the divergence is intentional and should be documented as such (e.g. "harmony arcs are deliberately more subtle than melody arcs, by design, because harmony is a bed not a foreground voice"). Either answer is legitimate — right now neither is written down anywhere, so it reads as an oversight rather than a decision, the same category of issue as item 4's motif-typing asymmetry.

### 9. `RhythmStrategyRegistry` and `MelodyStrategyRegistry` are dead code — finish the migration, but two of the dead classes need fixing first, not just reconnecting

- **Verified:** `strategies.py`'s module docstring (lines 1–20) describes a Strategy Pattern replacing "the if/elif/else switchboards in generate_section" for both rhythm and melody dispatch, and shows example usage: `RhythmStrategyRegistry.resolve(rhythm_source).apply(ctx)` and `MelodyStrategyRegistry.resolve(melody_beh).apply(mctx)`. `grep -rn "RhythmStrategyRegistry\|MelodyStrategyRegistry"` across the whole codebase turns up exactly two matches outside their own definitions — both are the docstring's own usage example, not executable code. `generator.py:49–50` imports both registries, but neither is ever called.
- **What's actually running instead:** the raw `if rhythm_source == "pattern": ... elif rhythm_source == "motif": ... else:` switch at `generator.py:547–584` — precisely the switchboard pattern the docstring claims was replaced. Only `HarmonyStrategyRegistry` made it to a real call site (`generator.py:1280`, `HarmonyStrategyRegistry.resolve(hrctx.source).apply(hctx)`) — the harmony migration finished, the rhythm/melody one didn't.
- **Decision (confirmed with the author):** finish the migration — the decoupling of melody/harmony/bass dependencies (the 2026-07 pass referenced throughout this doc) was a deliberate architectural choice, not a retreat from the Strategy Pattern. Abandoning the unused registries was never the intent; they were simply never reconnected after the decouple pass touched the code around them.
- **But three of the nine dead classes are not safe to reconnect as-is** — the Strategy Pattern classes predate the 2026-07 decouple pass and didn't get updated when it landed, so "just wire them in" would silently undo work that was deliberately done:
  - **`MotifRhythmStrategy` (rhythm.py source `"motif"`) still builds a coupled `bass_events`** from the motif's anchor grid (`strategies.py:411–427`, `bass_events = _motif_rhythm_to_events(rhythm, ..., "anchor", ...)`). This is exactly the coupling the live switchboard's own comment (`generator.py:568–578`) documents as removed on purpose: it used to make `rhythm: "motif"` silently override every `bass_style` except `"motif"` itself with anchor roots — e.g. `bass_style: "steady"` (used throughout `piece_broadway_boogie_v7.json`) would have silently stopped using its declared figure. **Fix before reconnecting:** strip the `bass_events` line, return `(mel_events, None)` — bass rhythm is now bass_style's job, full stop, matching the switchboard.
  - **`MotifMelodyStrategy` (label `"motif"`) and `RhythmicMelodyStrategy` (label `"rhythmic"`) reference `MelodyLiteral` values that no longer exist.** `MelodyLiteral` was deliberately narrowed to `lyrical, generative, sparse, develop` — `"motif"` and `"rhythmic"` were removed. These two classes are pure dead weight referencing a schema state that predates the narrowing. **Fix before reconnecting:** delete both classes and remove them from `MelodyStrategyRegistry`'s constructor list.
- **The other six are clean:** `PatternRhythmStrategy` and `FreeRhythmStrategy` match the live switchboard exactly. `LyricalMelodyStrategy`, `GenerativeMelodyStrategy`, `SparseMelodyStrategy`, `DevelopMelodyStrategy` are thin `_DelegatingMelodyStrategy` wrappers around `generate_melody_for_progression` — no evidence of drift, though worth a quick signature spot-check against `melody.py`'s current parameters before wiring, on general principle rather than any specific suspicion.
- **Fix, in order:**
  1. Patch `MotifRhythmStrategy` to drop the bass-anchor coupling.
  2. Delete `MotifMelodyStrategy` and `RhythmicMelodyStrategy`.
  3. Wire `generate_section`'s rhythm-source switch (547–584) through `RhythmStrategyRegistry`, and the melody-behavior dispatch through `MelodyStrategyRegistry`, matching the `HarmonyStrategyRegistry` precedent already proven correct in production.
  4. Render a known piece (e.g. `piece_broadway_boogie_v7.json`) before and after the swap and diff the MIDI output — it should be byte-identical, since the switchboard and the corrected registry should now do exactly the same thing. Any diff means something in steps 1–3 didn't match the live behavior and needs another pass before this ships.

### 10. No cross-section continuity for velocity arcs — every section's dynamic curve resets to `t=0`

- **Where:** `apply_velocity_arc` (`rhythm.py:826`) computes `t = i / max(n-1, 1)` from that section's own event list alone. There is no mechanism carrying the previous section's ending `arc_scale` forward as a starting condition for the next section's curve.
- **Effect:** even where two adjacent sections both use correctly-implemented arc values (unlike item 1's bug), a section ending on a `fade_out` tail and the next section starting a `swell` from its own `t=0` will still produce an audible seam — two independently-phased curves glued end to end, not a continuous shape. This is structurally the same issue as item 1 but persists even after item 1 is fixed.
- **Fix:** pass the previous section's final `arc_scale` into the next section's `apply_velocity_arc` call as a reference point, and blend the first bar or two of the new curve toward its own target shape instead of jumping straight to the type's fixed starting value. Lower priority than item 1 — item 1 is an outright bug with a known concrete instance; this is a smoothing improvement on top of correctly-working arcs.

### 11. `context.py`'s reactive cross-voice listening machinery should be pruned, not finished — it documents a design philosophy the author has deliberately moved past

- **This item is different in kind from every other item in this doc.** Everything else here is "finish an incomplete migration" or "fix a bug." This one isn't — the author has explicitly rejected the design goal this code was built for: *"'voices that listen to each other' — that's an outdated notion I have wanted to get away from. Coordination to time is good, but interdependency is bad."* That maps directly onto an existing project principle already in play elsewhere in this codebase (documented in the composer's own working notes): *"Ensemble coherence from scaffold, not shared onsets — voices should share tempo/meter/harmonic rhythm as scaffold; shared onset cell makes music rigid."* This file's reactive listening model is the same rigidity, applied to pitch/contour/density instead of onsets. The fix isn't to wire it up — it's to recognize it as vestigial and remove it, so the code matches the design philosophy the rest of the project has already moved to.
- **What's alive and legitimate — "coordination to time":** `PieceContext.suggest_transform()` (`context.py:359–422`), with its form-position weighting (`_FORM_WEIGHTS_OPENING`/`_MIDDLE`/`_ENDING`, 325–337) and usage-count penalty. This shapes motif transform choice based on *where a section sits in the piece's timeline* — not on what another voice just played. Confirmed as a real call site in `generator.py`. This is exactly "coordination to time" and should stay.
- **What's dead and shouldn't be revived — reactive "interdependency":** verified by tracing every actual generation function's signature, not assumed:
  - `generate_bass()`, `generate_melody_for_progression()`, `generate_counterpoint()` — **none of the three accept a `ctx`/`sec_ctx` parameter.** They cannot read a `VoiceSnapshot` even if one exists.
  - `SectionContext.add_voice()` is called four times in `generator.py` (bass, melody, each `voices[]` peer, each counterpoint voice) — write-only. `sec_ctx.get_voice()`, `.bass`, `.melody`, and `PieceContext.previous_melody`/`previous_bass` — the entire read-side API — have **zero call sites** anywhere in `generator.py` (grep-confirmed).
  - `generator.py`'s own comments assert a capability that doesn't exist: line 703's *"Record bass snapshot so melody/counterpoint can read it"* and line 715's section header *"MELODY — generates second, can read bass snapshot"* — the very next call (`generate_melody_for_progression`, line 730) doesn't pass `sec_ctx` at all. The comments describe the old, now-rejected design goal, not the actual code.
  - The specific dead surface: `VoiceSnapshot.pitch_center`, `.ending_contour`, `.rhythmic_profile`, `.achieved_density`, `.last_scale_degree` (all computed by `compute_voice_snapshot`/`compute_contour`/`compute_rhythmic_profile`), plus the read-side API listed above.
- **Not part of this — keep as-is:** `against_notes`/`against_voices`, passed directly into `generate_counterpoint` for parallel-fifths/octaves avoidance. This is a mechanical correctness constraint (raw pitch collision checking), not the reactive `VoiceSnapshot` musical-listening abstraction — different mechanism, different purpose, not something the "get away from interdependency" principle argues against.
- **Fix:**
  1. Delete the dead `VoiceSnapshot` fields (`pitch_center`, `ending_contour`, `rhythmic_profile`, `achieved_density`, `last_scale_degree`) and their compute helpers if nothing else uses them (`compute_contour`, `compute_rhythmic_profile` — check for other callers first).
  2. Delete `SectionContext.get_voice`/`.bass`/`.melody` and `PieceContext.previous_melody`/`previous_bass` — the unused read API.
  3. Fix the two misleading comments in `generator.py` (703, 715) so they don't describe a capability the code doesn't have.
  4. Rewrite `context.py`'s module docstring to state the actual, current philosophy — coordination to shared timing/form-position, not cross-voice reactive listening — so the next person reading this file (including a future session) doesn't reasonably conclude, as this session did, that reconnecting the dead read API is "finishing" something.
  5. Keep `VoiceSnapshot.last_transform`/`last_chord_degree` and everything `suggest_transform` actually consumes — those are the "coordination to time" parts, not interdependency.

## P2 — Minor / stylistic

### 12. `note_length_range` no-op conditions live in two places conceptually

`SectionModel.note_length_range`'s docstring (523–526) and `CounterpointModel.note_length_range`'s comment (277–280) both describe overlapping "ignored when groove is set / rhythm is pattern-or-motif" conditions, and both point to `lint.py` as the enforcer. Not wrong, just worth a single shared docstring/constant (`GRID_OVERRIDING_RHYTHM_SOURCES` already exists in `lint.py` for this) referenced from both schema docstrings instead of restated prose in each. (If item 6 above is implemented, this becomes three places instead of two — all the more reason to centralize the wording.)

### 13. `_check_bar_alignment`'s epsilon comparison

`schemas.py:857`: `remainder > 1e-6 and abs(remainder - self.beats_per_bar) > 1e-6` — correct, but the double-sided modulo-epsilon check is easy to misread. A short inline comment ("total must land on a bar boundary within floating-point tolerance, checked from both sides of the modulo wrap") would save the next reader (including future you) a beat of head-scratching.

### 14. `lint.py`'s `COUPLINGS` registry and `CHECKS` list are two hand-synced parallel structures

`lint.py`'s own docstring (18–23) spells out adding a coupling as three manual steps: document it in `COUPLINGS` (a plain-English string), write a `_check_*` function, add the function to `CHECKS`. Currently the two lists line up 1:1 (9 entries, 9 functions) — but nothing enforces that; a future check added without its `COUPLINGS` entry (or vice versa) would silently drift, same category of risk as `_KNOWN_KEYS` (#3) just smaller-blast-radius, since the cost of drift here is a stale doc string rather than a false validation error. Could derive `COUPLINGS` from each check function's docstring first line at import time instead of hand-duplicating the description in two places.

### 15. `motif_loader.py`: one dead function, one unnecessary defensive lazy-import, one cosmetic ordering slip

- **`save_motif` (101–127) is dead code.** `grep -rn "save_motif("` across the whole repo returns exactly one hit — its own definition. Not called anywhere, including the module's own `if __name__ == "__main__"` demo block, which exercises `list_available_motifs` and `load_motif` but never `save_motif`. Same "written, never wired up" pattern as `SongFormEntryModel.variation` and the unused strategy registries, just lower-stakes here since it's a self-contained write path rather than something with silent behavioral consequences.
- **`schemas.py`'s `_resolve_motif_value_safe` imports `motif_loader.resolve_motif_value` lazily inside the function body**, which normally signals a circular-import guard. Checked empirically: `motif_loader.py` only imports from `intervals.music.motif`, which only imports stdlib — no cycle exists. Confirmed by actually importing both `schemas.py` and `motif_loader.py` at top level in the same process; both load cleanly. The lazy import isn't load-bearing today — either it guarded against a circularity from an earlier version of `schemas.py` (which has clearly been restructured more than once, per its own inline history comments) that's since been resolved, or it was defensive caution nobody re-examined. **Fix:** hoist to a normal top-level `from intervals.core.motif_loader import resolve_motif_value` in `schemas.py`, and re-run the import check if paranoid — it should still pass.
- **Cosmetic:** the `# --- Quick test / demo ---` section-header comment (240–242) sits *after* the `if __name__ == "__main__"` block it appears to introduce, not before it — reads like a copy-paste/reorder slip from an earlier edit. No behavioral effect, just noted in passing.
- **Worth naming as a positive, not just findings:** `resolve_motif_value`'s three-way `None`/`str`/`dict` branch is the one place in this codebase that fully achieves the "single source of truth" goal the rest of the project aspires to but doesn't always hit — every schema-level motif override (`VoiceModel.motif`, `HarmonyRhythmModel.motif`, `SectionModel.motif`) routes through it, unlike the two-representations problem for motif *typing* (item 4) or the two divergent arc implementations (item 8). Motif *resolution* got this right; worth treating as the reference pattern when fixing the others.

## Authoring hygiene (not an engine bug — flagging from `piece_broadway_boogie_v7.json`)

`m2_accumulation` and `m4_peak_build_1`'s `notes` fields both describe `harmony=motif (stressed filter on the shaped v3 cell → five varied-length on-beat comp chords per 2-bar cycle...)`, but the actual `harmony_rhythm` block in both sections is `{"rhythm": "sustain", ...}` — sustained pads, not motif-driven comping. The documentation and the field disagree, in the same way in both sections, which suggests a field was changed in an earlier edit pass without updating the accompanying note. Worth deciding which one is true (revert the field to `"motif"`, or update the notes) — not something `lint.py` can catch generically since it's free-text vs. a typed field, but worth a manual pass over the piece file.

## Suggested order of attack

1. **Fix the harmony arc curve gap (#1) first** — `build`/`decay`/`fade` are genuinely missing, and the correct math already exists in `velocity_envelope`; this is the only item in the list with a confirmed, traced connection to an audible artifact.
2. Decide `variation`'s fate (#2, kill it or wire it) — the only *other* field currently living in a blind spot relative to your own lint discipline.
3. `_KNOWN_KEYS` → derived from `model_fields` (#3, mechanical, ~10 min, removes a recurring maintenance tax).
